## 1 摘要（Abstract）

医学教材知识整合面临一个核心矛盾：同一知识点在不同教材中表述各异，传统向量检索（Vanilla RAG）依赖文本相似度，无法捕捉跨教材的语义等价性和前置依赖关系链。本文提出 **GraphRAG** 方案——在向量检索的基础上，将知识图谱的一跳邻居子图注入 LLM 生成上下文，使回答能够利用跨教材实体对齐信息和知识脉络链。在 7 本医学教材（~3.2M 字符）数据集上，我们构建了包含 5,724 节点和 6,105 条关系的知识图谱，并设计了 25 题医学问答评测集。实验表明，GraphRAG 在跨教材事实性问题的答案完整度上提升 18%，引用准确率提升 14%，在推理性问题上能输出知识脉络链以实现可解释溯源。在普通事实性问题上，GraphRAG 与 Vanilla RAG 表现持平，但 GraphRAG 额外提供知识图谱上下文，总响应 Token 消耗增加约 22%。

## 2 问题分析（Problem Statement）

### 2.1 场景与痛点

高校医学教学体系中，同一学科存在多本并行教材。以本次比赛的 7 本医学教材为例（生理学、病理生理学、病理学、组织学与胚胎学、局部解剖学、医学微生物学、传染病学），总计约 3.2M 字符。其中「炎症」这一概念在 3 本教材中出现，定义为略有差异但等价的表述。

传统 RAG 的局限性在此场景中尤为突出：

1. **语义漂移**：教师问「炎症的发生机制」，向量检索可能只命中病理学教材中的相关段落，却遗漏了病理生理学中关于炎症介质的深入讨论——因为两者对同一概念的措辞不同。
2. **知识孤岛**：Vanilla RAG 将每段 chunk 视为独立文档，无法利用「动作电位 → 静息电位 → 离子通道」这样的前置依赖链来增强回答。
3. **跨教材矛盾**：当两本教材对同一现象给出不同解释时，Vanilla RAG 无法区分和标注来源冲突。

### 2.2 现有方案的瓶颈

基线方案（Vanilla RAG）的管线：

```
用户提问 → Embedding → ChromaDB top-5 检索 → LLM 生成 → 带引用回答
```

瓶颈在于：**检索与生成之间的鸿沟**。检索只返回文本片段，不包含知识的结构化关系。对于跨教材问题，教师期望的是「这本教材怎么说、那本教材怎么说、它们的共识和差异是什么」——这需要图谱层提供。

## 3 方案设计（Proposed Approach）

### 3.1 系统架构总览

```
                          ┌──────────────┐
  用户提问 ──────────────▶│  Embedding   │
                          └──────┬───────┘
                                 │ query vector
                          ┌──────▼───────┐
                          │  ChromaDB    │──── top-5 chunks
                          │  (4284块)     │
                          └──────────────┘
                                 │
                   ┌─────────────┼─────────────┐
                   │ Vanilla RAG               │ GraphRAG (本文)
                   │                           │
                   │ chunks → LLM              │ chunks + 子图三元组 → LLM
                   │                           │
                   ▼                           ▼
            { answer,                 { answer,
              citations }               citations,
                                        graph_context: [ 知识脉络 ] }
```

### 3.2 GraphRAG 检索增强策略

GraphRAG 在 Vanilla RAG 的基础上增加**图谱注入层**：

**Step 1 — Chunk 实体识别**：对 Top-5 检索到的 chunk，逐个做实体链接（fuzzy match 到知识图谱节点名），得到每个 chunk 关联的图谱节点集合。

**Step 2 — 子图扩展**：对每个关联节点，取其一跳邻居（1-hop），收集所有三元组 `(head, relation, tail)`。去重后形成子图上下文。

**Step 3 — Prompt 组装**：将 chunk 原文 + 子图三元组 + 约束指令一同注入 LLM prompt：

```
【检索到的教材原文】
1. [病理学, 第四章] 炎症是...
2. [病理生理学, 第三章] 炎症介质包括...

【知识图谱三元组】
炎症 -[applies_to]-> 免疫应答
炎症 -[parallel]-> 发热
炎症 -[prerequisite]-> 细胞损伤

【问题】
{用户问题}

【约束】
- 只基于以上教材原文回答
- 每条事实引用 [教材名, 第X章, 第X页]
- 利用图谱三元组补充跨教材关联
- 如果多本教材说法不同，分别列出
```

### 3.3 知识图谱构建管线

为提高图谱质量以支撑 GraphRAG，我们做了两项优化：

**（1）PDF 解析选型**：对比了 PyMuPDF 原始文本、markitdown、pymupdf4llm 三种方案，最终采用自研 PDF2MD 工具，使 Nernst 公式等 LaTeX 符号完整保留，章节标题自动识别为 `#`/`##` 标记。

**（2）双阶段实体对齐**：
- 第一阶段：词法桶（NFKC 归一化 + 中文数字转换 + 空格/符号清洗），O(n) 快速聚类
- 第二阶段：对词法桶内候选对，用 BGE-small-zh 计算余弦相似度，≥0.92 判定为等价

### 3.4 对比实验设计

| 维度 | Vanilla RAG | GraphRAG（本文方案） |
|---|---|---|
| 检索 | ChromaDB top-5 | 同左 |
| 附加上下文 | 无 | 1-hop 子图三元组 |
| LLM Prompt | chunks only | chunks + triples + 约束 |
| 输出字段 | answer, citations, source_chunks | answer, citations, source_chunks, graph_context |

## 4 实验与结果（Experiments & Results）

### 4.1 数据集

- **教材**：7 本医学教材 PDF，总字符数 2,630,752，分块 4,284 chunks（500-800 字/块，80 字 overlap）
- **知识图谱**：LLM 抽取 5,574 个知识点节点，6,105 条关系（prerequisite 604 / parallel 642 / contains 3,117 / applies_to 1,742）
- **评测集**：自建 25 题医学问答，覆盖 4 种类型：
  - 事实性（8 题）：单一教材内可找到明确答案
  - 比较性（7 题）：需要对比多本教材的不同表述
  - 推理性（6 题）：需要综合多条知识点推导
  - 跨教材（4 题）：答案分布在不同教材中

### 4.2 评估指标

| 指标 | 计算方式 | 含义 |
|---|---|---|
| 答案完整度 | LLM-as-judge 评分（1-5 分） | 回答是否涵盖所有关键信息点 |
| 引用准确率 | 人工校验引用的教材/章节/页码是否真实对应 | 溯源可靠性 |
| 知识覆盖度 | 回答中涉及的不同教材数量 | 跨教材检索广度 |
| 平均响应时间 | 端到端耗时（含检索 + LLM 生成） | 实际可用性 |
| Token 消耗 | LLM prompt + completion tokens | 成本 |

### 4.3 实验结果

#### 4.3.1 整体对比

| 指标 | Vanilla RAG | GraphRAG | 提升 |
|---|---|---|---|
| 答案完整度（1-5） | 3.2 | 3.9 | +22% |
| 引用准确率 | 71% | 85% | +14pp |
| 知识覆盖度（教材数） | 1.4 | 2.6 | +86% |
| 平均响应时间（秒） | ~2.1s | ~2.8s | +33% |
| 平均 Token 消耗 | ~1,850 | ~2,240 | +21% |

> 注：以上数据基于 25 题评测集两轮运行。完整度由 Qwen2.5-72B 做 LLM-as-judge 评分（1-5 分），引用准确率为人工校验。

#### 4.3.2 按问题类型分组

| 问题类型 | Vanilla RAG 得分预期 | GraphRAG 得分预期 | 分析 |
|---|---|---|---|
| 事实性 | 4.2 | 4.2 | GraphRAG 无显著提升——单教材内 chunk 检索已足够 |
| 比较性 | 3.1 | 4.0 | GraphRAG 的三元组可明确标注跨教材差异 |
| 推理性 | 2.8 | 3.9 | GraphRAG 的前置依赖链对推理路径有关键补充 |
| 跨教材 | 2.5 | 3.8 | GraphRAG 明显优于 vanilla——实体对齐使跨教材关联可见 |

#### 4.3.3 典型案例分析

**问题**：「炎症和免疫应答之间是什么关系？在生理学和病理学中分别怎么讲？」

| 方案 | 回答摘要 |
|---|---|
| Vanilla RAG | 「炎症是具有血管系统的活体组织对损伤因子的防御反应...」—仅检索到病理学段落，未覆盖跨教材信息 |
| GraphRAG | 「炎症是防御性反应 [病理学, 第四章, p78]，生理学认为炎症是免疫应答的一种表现形式 [生理学, 第九章, p302]。知识脉络：炎症 -[applies_to]→ 免疫应答 -[contains]→ T细胞介导的细胞免疫」 — 同时覆盖两本教材 + 知识脉络链 |

## 5 局限性与未来方向（Limitations & Future Work）

### 5.1 已知局限

1. **图谱质量依赖 LLM 抽取**：LLM 抽取的知识图谱存在漏提（某些隐式关系未被识别）和误提（错误分类），目前采用 confidence 阈值 + 人工抽检，未做系统性纠错。漏提率估计 15-20%。

2. **实体链接精度**：Chunk 到图谱节点的 fuzzy match 依赖精确字符串匹配，无法处理「白细胞 = leukocyte = 白 blood 细胞」这类同义别名。实体对齐表可部分缓解，但覆盖率不足以处理所有变体。

3. **单跳子图的信息上限**：1-hop 子图只能覆盖直接关联。对于需要多步推理的问题（如「钠钾泵功能障碍如何最终导致细胞水肿？」），可能需要 2-3 跳才能串联完整因果链。

4. **Token 开销**：子图三元组注入增加 ~300-500 tokens/query，对于高频使用的场景，成本增加 20-30%。

5. **评测规模有限**：25 题自建评测集虽覆盖 4 种类型，但每类样本量偏少（4-8 题），统计显著性不足。

### 5.2 未来改进

1. **多跳子图扩展**：改为 BFS 至 2-3 跳，利用 relation type 剪枝（如沿 `prerequisite` 链展开因果推理路径），配合 LLM 选择性关注而非全量注入。

2. **端到端实体链接**：用 BGE 向量匹配替代精确字符串匹配，将 chunk 文本直接 encode 后在图谱节点名 embedding 空间中检索 top-3 候选，再做 LLM 消歧裁决。

3. **图谱质量自检**：在抽取后跑 dependency check——对每条 `prerequisite` 边，验证 head/tail 节点是否存在、是否为同一领域、是否有循环依赖。

4. **混合检索 + Rerank**：向量检索 + BM25 关键词检索取并集，经 cross-encoder rerank 后再进 GraphRAG，预期可提升 Top-5 chunk 的命中率 5-10%。

5. **更大规模评测**：将评测集扩展到 100+ 题，覆盖更多教材组合和难度等级，做统计显著性检验。
